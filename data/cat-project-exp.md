## 2. 描述一个你解决过的最难的技术问题？

> **难度**: hard | **分类**: 项目经验 | **ID**: 98

#### 🎯 本质
面试官通过此题考察**问题分析能力、技术深度、方法论**，而非单纯看你解决过什么。

#### 🧒 类比
就像医生讲一个疑难病例——不是炫耀手术刀法，而是展示**诊断思路**（问诊→检查→排除→确诊→治疗）。

#### 📊 图解
STAR法则讲述框架：

┌──────────────────────────┐
│ S 背景：问题发生在什么场景  │
│ T 任务：你负责什么         │
│ A 行动：你做了哪些关键步骤  │
│ R 结果：量化成果+经验总结   │
└──────────────────────────┘

关键比例：背景20% + 分析40% + 方案30% + 总结10%
#### 🔧 详解
回答结构：
**1.问题背景(20%)**：简述场景，突出难度（不是加个按钮那么简单）
**2.排查过程(40%)**：展示系统性思维——从现象→假设→验证→定位。提工具名（Chrome DevTools、WireShark、perf）
**3.解决方案(30%)**：讲为什么选这个方案（对比过几个方案），关键代码/配置改动
**4.结果(10%)**：量化数据（性能提升X%，崩溃率从X降到Y）+ 经验总结

#### 💻 代码
// 模板示例
\"我们的SPA首屏加载需要4s+，用户投诉率高。\"

排查：
1. Lighthouse分析 → LCP 3800ms
2. Network瀑布图 → vendor.js 1.8MB阻塞
3. Source map → moment.js占600KB(仅用3个方法)

方案：
1. 路由懒加载(减少首屏体积)
2. moment → dayjs(体积减96%)
3. CDN + 资源预加载

结果：LCP从3.8s→1.2s，跳出率降15%
#### ❓ 追问
如果让你重新做，会怎么改进？遇到过类似问题吗？如果排查方向错了怎么办？

## 3. 介绍一个你最满意的项目？

> **难度**: medium | **分类**: 项目经验 | **ID**: 463

#### 🎯 本质
选择**有技术深度+有业务价值**的项目。讲清楚难点和你的解决方案。

#### 🧒 类比
介绍项目像推荐好电影——有剧情(背景)、有高潮(难点)、有结局(成果)。

#### 📊 图解
要点:
1.项目背景(1句)
2.技术栈
3.你的角色和职责
4.核心难点+解决方案
5.量化成果
#### 🔧 详解
选一个你能深入聊的项目。说明你负责的部分。重点讲遇到的技术难点和解决方案(面试官会追问细节)。成果要量化。准备好被问'为什么选这个方案'。

#### 💻 代码
// 框架
'XX项目是一个日活X万的XX应用。
技术栈: Vue3+TypeScript+Pinia+Vite
我负责: 核心业务模块+性能优化
难点: 首屏加载慢/大数据渲染卡顿
方案: SSR+虚拟列表+按需加载
成果: 首屏从3秒降到0.8秒，用户留存提升20%'
#### ❓ 追问
面试官追问细节怎么办？答：提前准备好技术选型理由和备选方案对比。

## 4. 项目中遇到的最大困难？

> **难度**: medium | **分类**: 项目经验 | **ID**: 464

#### 🎯 本质
描述**困难→分析→解决→收获**的过程。体现问题解决能力。

#### 🧒 类比
讲困难像讲故事——遇到怪兽(困难)、分析弱点(根因)、用武器(方案)打败、获得经验值(收获)。

#### 📊 图解

```
困难→分析→尝试→解决→收获
```

#### 🔧 详解
选一个真实的技术困难。描述当时的现象(表面问题)。你的分析过程(如何定位根因)。尝试的方案(为什么选这个)。最终解决的结果。你学到了什么。

#### 💻 代码
// 示例
'项目上线后偶发白屏，排查发现是某个接口超时导致JS错误。
分析: 监控发现特定机型+弱网触发，接口返回HTML被当JS执行。
方案: 接口加超时降级+responseType校验+全局错误边界。
收获: 学会了系统化排查线上问题，建立监控体系。'
#### ❓ 追问
不要说什么困难都没遇到。选一个能体现你能力的。

## 5. 如何做技术选型？

> **难度**: hard | **分类**: 项目经验 | **ID**: 465

#### 🎯 本质
技术选型要考虑**团队能力、项目需求、生态成熟度、长期维护**。

#### 🧒 类比
技术选型像买工具——不是最贵最好，而是最适合的。

#### 📊 图解
评估维度:
1.项目需求匹配
2.团队能力
3.社区生态
4.学习曲线
5.长期维护
#### 🔧 详解
先明确需求(性能要求/团队规模/时间约束)。然后从需求出发评估候选方案。不要只看GitHub Star，看实际使用案例和活跃度。考虑团队现有技术栈(迁移成本)。关注长期维护(社区活跃/版本更新)。

#### 💻 代码
// 选型评估表
const evaluation={
  React: {teamSkill:9,ecosystem:10,performance:8,suitability:9},
  Vue:   {teamSkill:7,ecosystem:9,performance:8,suitability:8},
  Svelte:{teamSkill:3,ecosystem:6,performance:9,suitability:7}
};
// 最终选React: 团队熟悉+生态最好+满足性能需求
#### ❓ 追问
选错了怎么办？答：及时止损+做好迁移计划。如何说服团队？答：Demo+数据+风险分析。

## 6. 你是如何保持技术学习的？

> **难度**: medium | **分类**: 项目经验 | **ID**: 466

#### 🎯 本质
展示**学习方法+持续动力+实际应用**。不只是看还要做。

#### 🧒 类比
学习像健身——持续比强度重要，理论+实践结合。

#### 📊 图解
方法:
1.官方文档+源码
2.技术博客/社区
3.个人项目实践
4.技术分享
5.关注前沿
#### 🔧 详解
学习方法：官方文档是第一优先。阅读优秀开源项目源码。关注技术社区(HN/掘金/InfoQ)。个人项目实践新技术。团队内做技术分享。学习要有输出(博客/项目)。

#### 💻 代码
// 学习路线示例
'我每周花5小时学习：
1.读官方文档和RFC(2小时)
2.做side project实践(2小时)
3.写技术博客输出(1小时)
最近在学Rust，做了一个CLI工具体会所有权机制。
学习新技术我会先看文档→做小项目→读源码→写博客总结。'
#### ❓ 追问
你的技术学习来源？答：官方文档>源码>技术博客>视频教程。
